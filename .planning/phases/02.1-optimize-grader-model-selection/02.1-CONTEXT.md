<decisions>
## Implementation Decisions

### Benchmarking approach
- Two-stage process: fast API-level benchmark script first, then `test:bootstrap` for final e2e validation of top candidates
- Direct Ollama `/api/generate` calls in the benchmark script (not through the grader code)
- 3 runs per model per configuration to get stable medians
- Pass/fail criteria: JSON parse success >= 100%, wall time <= 60s, score > 0.0, score plausibility (manual sanity check). Ranking among passing models: fastest wins, tie-break by higher score
- Permanent test fixture in `tests/` — reusable for future model evaluations
- Console table output during run (instant feedback) plus JSON results file
- Benchmark report committed as markdown in the phase directory (BENCHMARK.md)
- Test data: one real session log captured from `test:bootstrap` as fixture, plus two synthetic negative cases (empty agent output and wrong solution)
- Single rubric: superlint `quality.md` for all benchmark runs
- Pull all candidate models upfront before benchmarking starts

### Model candidates
- Start with Phase 2 research candidates: phi3.5:3.8b (incumbent), qwen2.5:3b, llama3.2:3b, qwen2.5:7b
- Add gemma3:4b (user requested)
- Full phase research to survey additional candidates based on use case (JSON structured output for LLM-as-judge rubric grading) and hardware constraints (16GB RAM, ARM64 CPU-only, 60s budget)
- Two size tiers: 3-4B (primary) and 7B (quality upgrade). Report results separately
- Single model selected for both local and CI platforms
- JSON Schema structured output tested per model — if a model fails with JSON Schema, fall back to no-schema mode and record both results
- Research which quantization variants make sense to benchmark per model — don't assume only Q4_K_M and Q5_K_M. Measure, don't assume speed/quality claims from prior research
- Phase 2 supplementary research is grounding material, not authoritative — run proper gsd-phase-researcher cycle

### Ollama tuning scope
- Both layers in scope:
  - Layer 1: Ollama serve environment vars (FLASH_ATTENTION, KV_CACHE_TYPE, NUM_PARALLEL, NUM_THREAD)
  - Layer 2: API request params (JSON Schema format, num_ctx, num_predict, num_thread, num_batch, num_gpu, temperature)
- Before/after benchmarking to measure actual tuning impact (3-point matrix):
  1. Default Ollama + default API params
  2. Optimized env vars + default API params
  3. Optimized env vars + optimized API params
- Isolates env var impact (1 vs 2), API param impact (2 vs 3), and combined impact (1 vs 3)
- Wrapper shell script orchestrates Ollama lifecycle (start/stop/restart with different env vars). Must work in Git Bash on Windows
- JSON Schema structured output applied unconditionally as the default format; no-schema is the fallback for models that fail with it
- Programmatic validation: LLMGrader warns (not fails) when suboptimal Ollama settings detected. Research phase determines detection mechanism (Ollama API inspection vs env var check)
- Document env vars in README.md
- Add env vars to ci.yml test-integration job
- Clean up GraderConfig: remove `num_ctx` and `timeout_ms` (Ollama-specific leaks). Ollama params live in `callOllama()` with benchmark-validated defaults
- Specific tuning values (num_ctx, num_predict, num_thread, etc.) determined by benchmark results, not hardcoded from prior research

### CI verification strategy
- Lightweight manual-dispatch CI benchmark workflow (`.github/workflows/benchmark-grader.yml`)
- `workflow_dispatch` trigger with model list and profile inputs
- Temporarily add `pull_request` trigger during Phase 2.1 development for auto-running; remove before phase completes
- Use `gh api` to dispatch on feature branch before merging
- Runs on `ubuntu-24.04-arm` runners
- Both tuning profiles (default + optimized) tested on CI, not just optimized
- Research `ai-action/setup-ollama@v2` — verify ARM64 support, env var configuration, version pinning. Then verify it works in our workflow
- CI results uploaded as artifacts; I (Claude) do the manual comparison of CI vs local results and document findings in the benchmark report
- Benchmark script calling `/api/generate` directly IS the "direct Ollama API call" verification from the success criteria

### Claude's Discretion

- Console output formatting and progress reporting
- Wrapper script cross-platform process management (stopping Ollama on Windows vs Linux)
- How to detect Ollama configuration at runtime (if research identifies a reliable method)
- Order of benchmark execution (models first vs profiles first)
