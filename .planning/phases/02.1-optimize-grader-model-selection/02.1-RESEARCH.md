# Phase 2.1: Optimize Grader Model Selection - Research

**Researched:** 2026-03-09
**Domain:** Local LLM grader model benchmarking, Ollama CPU optimization, CI ARM64 runners
**Confidence:** HIGH

## Summary

This phase requires building a benchmark harness to systematically evaluate grader model candidates on two ARM64 platforms (local Snapdragon X Elite with 32 GB RAM and GitHub's ubuntu-24.04-arm Cobalt 100 runners with 4 vCPU / 16 GB RAM), applying Ollama environment and API tuning, verifying through direct Ollama `/api/generate` calls and the e2e bootstrap test, and cleaning up GraderConfig. The existing Phase 2 supplementary research provides grounding material but its claims (phi3.5:3.8b at 14s, env var impact estimates) must be verified through actual benchmarking -- that is the core purpose of this phase.

Five model candidates are locked: phi3.5:3.8b (incumbent), qwen2.5:3b, llama3.2:3b, gemma3:4b (user-requested), and qwen2.5:7b. All have Q4_K_M quantization available on Ollama; phi3.5, qwen2.5, and llama3.2 also have Q5_K_M. Gemma3:4b adds a QAT (Quantization-Aware Training) variant. The benchmark must test JSON Schema structured output per model and fall back to no-schema mode on failure. A three-point tuning matrix (default/default, optimized-env/default-api, optimized-env/optimized-api) isolates the impact of each tuning layer.

The CI benchmark workflow uses `ai-action/setup-ollama@v2` (defaults to Ollama v0.17.7) on `ubuntu-24.04-arm` runners (4 vCPU Cobalt 100, 16 GB RAM for this public repo). The wrapper script must manage Ollama process lifecycle cross-platform (Git Bash on Windows vs native Linux in CI), using `ollama serve &` with PID tracking and platform-conditional kill commands.

**Primary recommendation:** Build the benchmark script as a standalone TypeScript file using native `fetch()` against `/api/generate`, reusing the existing prompt construction pattern from `LLMGrader`. Orchestrate tuning profiles via a shell wrapper that starts/stops Ollama with different environment variables. Produce JSON results plus a committed BENCHMARK.md report.

### Pattern 2: Three-Point Tuning Matrix via Shell Wrapper
**What:** A bash script that starts/stops Ollama with different environment variable profiles, then invokes the TypeScript benchmark script for each.
**When to use:** Local benchmarking to isolate env var impact vs API param impact.
**Example:**
```bash
#!/usr/bin/env bash
# scripts/ollama-bench.sh -- Cross-platform Ollama benchmark orchestrator

stop_ollama() {
  if [[ "$OSTYPE" == msys* ]] || [[ "$OSTYPE" == mingw* ]]; then
    taskkill.exe //IM ollama.exe //F 2>/dev/null || true
  else
    pkill ollama 2>/dev/null || true
  fi
  sleep 2
}

wait_for_ollama() {
  local retries=30
  while ! curl -sf http://localhost:11434/ > /dev/null 2>&1; do
    retries=$((retries - 1))
    if [[ $retries -le 0 ]]; then
      echo "[ERROR] Ollama failed to start within 30s"
      exit 1
    fi
    sleep 1
  done
}

# Profile 1: Default

ollama serve &
OLLAMA_PID=$!
wait_for_ollama
npx ts-node tests/benchmark-grader.ts --profile default --output results-default.json

OLLAMA_NUM_PARALLEL=1 \
OLLAMA_NUM_THREAD=12 \
ollama serve &
OLLAMA_PID=$!

stop_ollama
```

### Pattern 3: JSON Schema Structured Output
**What:** Pass a JSON Schema object as the `format` field in `/api/generate` requests. Ollama compiles it to a GBNF grammar that constrains token sampling.
**When to use:** Default for all grading calls. Fall back to no-schema if a model fails.
**Example:**
```json
{
  "type": "object",
  "properties": {
    "score": { "type": "number", "minimum": 0.0, "maximum": 1.0 },
    "reasoning": { "type": "string" }
  },
  "required": ["score", "reasoning"]
}
```
**Confidence:** HIGH -- Verified via [Ollama structured output docs](https://docs.ollama.com/capabilities/structured-outputs). GBNF grammar masks illegal tokens at sampling time; structurally cannot produce non-conforming JSON. The `minimum`/`maximum` constraints on numeric fields are supported by Ollama's JSON Schema implementation.

### Pattern 4: Ollama Configuration Detection at Runtime
**What:** Use the `/api/show` and `/api/ps` endpoints to inspect the running Ollama server and loaded models.
**When to use:** LLMGrader warning when suboptimal settings are detected.
**Detection strategy:**
- `/api/show` returns `details.quantization_level` and model parameters but does **not** expose runtime env vars like `OLLAMA_FLASH_ATTENTION`
- `/api/ps` shows running models with `size_vram`, `context_length`, and processor allocation (CPU/GPU split) but does **not** expose `num_thread` or KV cache type
- **There is no Ollama API endpoint that exposes server-level environment variables**

**Recommended approach for LLMGrader warning:**
The only reliable detection method is to check environment variables from the Node.js process side:
```typescript
function checkOllamaConfig(): string[] {
  const warnings: string[] = [];
  // These env vars must be set BEFORE ollama serve starts
  // We can only check if they're in our own process environment
  // (which indicates the user intended to set them)
  if (!process.env.OLLAMA_FLASH_ATTENTION) {
    warnings.push('OLLAMA_FLASH_ATTENTION not set -- KV cache quantization disabled');
  }
  if (!process.env.OLLAMA_KV_CACHE_TYPE) {
    warnings.push('OLLAMA_KV_CACHE_TYPE not set -- using FP16 KV cache (higher RAM usage)');
  }
  return warnings;
}
```
**Confidence:** MEDIUM -- This only detects env vars in the current process, not in the Ollama server process. If Ollama is started separately (common on Windows desktop), these checks may produce false warnings. The planner should consider whether to check `process.env` only or skip runtime detection entirely and rely on documentation.

### Anti-Patterns to Avoid
- **Benchmarking through `LLMGrader`:** The grader adds retry logic, fallback chains, and parse error handling that mask raw model performance. Benchmark directly against `/api/generate`.
- **Trusting prior research claims without measurement:** The Phase 2 research claims phi3.5 at 14s and env var speedups. These must be verified by this phase's benchmarks, not assumed.
- **Hardcoding tuning values from prior research:** Values like `num_ctx: 8192`, `num_thread: 12`, `num_batch: 128` come from Phase 2 research. The benchmark must determine optimal values empirically.
- **Running Gemma 3 4B with KV cache quantization on old Ollama versions:** Issue [#9683](https://github.com/ollama/ollama/issues/9683) reported extreme slowdown with `OLLAMA_KV_CACHE_TYPE=q8_0` on Gemma 3. The issue was resolved in Ollama versions after v0.6.0 (the issue was closed October 2025). With Ollama v0.17.7 (the CI default), this should be safe, but the benchmark will detect any regression.
- **Using `format: "json"` string instead of JSON Schema object:** The string form only enforces valid JSON syntax, not the grading schema. The model may produce arbitrary field names.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON schema enforcement | Regex parsing + retry loop | Ollama `format` with JSON Schema object | GBNF grammar constrains at token level; structurally impossible to produce non-conforming output |
| ARM64 core optimization | Custom thread pinning | `OLLAMA_NUM_THREAD` env var | Ollama's built-in threading with explicit count works around the ARM64 detection bug |
| CI Ollama installation | Manual curl + extract | `ai-action/setup-ollama@v2` | Handles platform detection, version pinning, PATH setup |
| Model pulling in CI | Custom pull script | `ollama pull` in workflow step | Built-in progress, retry, and verification |
| Benchmark statistics | Custom median/percentile | Sort + index middle | Only need median of 3 runs; no stats library required |

**Key insight:** The benchmark script is simple fetch-parse-measure code. The complexity is in the orchestration (shell wrapper managing Ollama lifecycle across profiles) and the analysis (interpreting results across models x quantizations x profiles x platforms).

## Common Pitfalls

### Pitfall 1: Ollama Not Ready After Serve Start
**What goes wrong:** Benchmark script tries to call `/api/generate` before Ollama has finished initializing, getting ECONNREFUSED.
**Why it happens:** `ollama serve &` returns immediately but the server takes 1-5 seconds to bind the port and become ready.
**How to avoid:** Poll `curl -sf http://localhost:11434/` in a retry loop with 1s delay, max 30 retries, before running the benchmark.
**Warning signs:** Intermittent ECONNREFUSED on first model, especially on cold starts.

### Pitfall 2: Model Cold-Load Penalty Skewing First Run
**What goes wrong:** The first benchmark run for a model is 2-5x slower than subsequent runs because Ollama must load the model into memory.
**Why it happens:** Ollama lazy-loads models on first request. Subsequent requests reuse the loaded model (kept alive for 5 minutes by default).
**How to avoid:** Send a warm-up request before the timed runs. Or: pull all models upfront (as decided), and send one throwaway generate request per model before the 3 measured runs.
**Warning signs:** Run 1 is dramatically slower than runs 2-3 for the same model.

### Pitfall 3: Model Swap Overhead Between Candidates
**What goes wrong:** When switching from Model A to Model B, the first request to Model B triggers unloading Model A and loading Model B, adding 5-15 seconds of overhead.
**Why it happens:** `OLLAMA_NUM_PARALLEL=1` and limited RAM means only one model can be loaded at a time. The swap happens transparently on the first request.
**How to avoid:** Include a warm-up request for each model switch. The benchmark script should detect model transitions and insert a warm-up call. Do not count the warm-up in timing.
**Warning signs:** First run of each new model is slower than expected; subsequent same-model runs are fast.

### Pitfall 4: Gemma 3 KV Cache Quantization Performance
**What goes wrong:** Gemma 3 runs extremely slowly with `OLLAMA_KV_CACHE_TYPE=q8_0`.
**Why it happens:** Ollama issue [#9683](https://github.com/ollama/ollama/issues/9683) reported this on v0.6.0. The issue was closed/resolved in October 2025.
**How to avoid:** The CI uses Ollama v0.17.7 (well past the fix). The benchmark will naturally detect this if it regresses. If gemma3:4b is anomalously slow in the "optimized-env" profile vs "default", investigate KV cache as root cause.
**Warning signs:** gemma3:4b taking 5-10x longer in optimized-env profile compared to default.

### Pitfall 5: Git Bash Process Management Differences
**What goes wrong:** The wrapper script's `stop_ollama()` function fails to kill Ollama on Windows.
**Why it happens:** Git Bash `kill` works with PIDs from the bash session, but `ollama serve` on Windows may spawn a child process that outlives the parent. `pkill` is not available in Git Bash.
**How to avoid:** On Windows (MSYS/MINGW), use `taskkill.exe //IM ollama.exe //F` (note double-slash for Git Bash flag escaping). On Linux, use `pkill ollama`. Check `$OSTYPE` to select the right method.
**Warning signs:** Port 11434 still in use after stop_ollama(); "address already in use" errors when restarting.

### Pitfall 6: CI Runner RAM Constraint (16 GB)
**What goes wrong:** qwen2.5:7b with high num_ctx causes OOM on CI runners.
**Why it happens:** 16 GB RAM minus OS overhead leaves ~14 GB. qwen2.5:7b Q4_K_M is ~4.7 GB for weights; at 8K context with FP16 KV cache, total could be ~6-7 GB. With q8_0 KV cache, it fits comfortably, but at 16K+ context it may not.
**How to avoid:** Cap `num_ctx` at 8192 for 7B models on CI. Monitor for OOM via Ollama error responses (HTTP 500 or process crashes).
**Warning signs:** Ollama process killed by OOM killer; sudden benchmark failures on 7B models only.

### Pitfall 7: Phi 3.5 Default Tag Uses Q4_0, Not Q4_K_M
**What goes wrong:** Benchmark reports "phi3.5:3.8b" performance but is actually running Q4_0 quantization, not Q4_K_M.
**Why it happens:** The `phi3.5:latest` and `phi3.5:3.8b` tags on Ollama default to Q4_0 (2.2 GB), not Q4_K_M (2.4 GB). This differs from llama3.2 and qwen2.5 which default to Q4_K_M.
**How to avoid:** Use fully-qualified tags for benchmarking: `phi3.5:3.8b-mini-instruct-q4_K_M`. Record the actual quantization in results by calling `/api/show` and reading `details.quantization_level`.
**Warning signs:** phi3.5 model size is 2.2 GB instead of 2.4 GB; `quantization_level` shows Q4_0.

## Open Questions

1. **Optimal num_thread for Snapdragon X Elite mixed cores**
   - What we know: 12 total cores (mixed performance + efficiency). Phase 2 research used 12 as default.
   - What's unclear: Whether using only performance cores (8 threads) is faster due to memory bandwidth. The benchmark will test 8, 10, and 12.
   - Recommendation: Include num_thread variations as a sub-matrix in the benchmark. Keep the matrix small (3 values).

2. **Gemma 3 4B QAT vs Q4_K_M quality difference**
   - What we know: QAT models are trained with quantization awareness, theoretically preserving more quality than post-training quantization. The QAT variant is 4.0 GB vs Q4_K_M at 3.3 GB.
   - What's unclear: Whether the quality difference is measurable in grading output (as opposed to perplexity benchmarks).
   - Recommendation: Benchmark both; the score comparison across the 3 test fixtures will reveal quality differences.

3. **Whether ai-action/setup-ollama@v2 works on ubuntu-24.04-arm**
   - What we know: The action downloads the correct Ollama binary for the platform. Ollama releases include ARM64 Linux builds. The action's README only shows `ubuntu-latest` examples.
   - What's unclear: Whether the action has been tested specifically on `ubuntu-24.04-arm` runners. There may be issues with `zstd` extraction or PATH setup.
   - Recommendation: Test early in a CI run before building the full benchmark workflow. If it fails, fall back to manual `curl` install of the ARM64 Ollama binary.

4. **Ollama configuration detection reliability**
   - What we know: No Ollama API endpoint exposes server-level env vars. `/api/ps` shows model state but not `num_thread` or KV cache type. `/api/show` shows model metadata but not runtime config.
   - What's unclear: Whether checking `process.env` from Node.js is a reliable proxy for the Ollama server's actual configuration.
   - Recommendation: The LLMGrader warning should check `process.env` and note in the warning message that these env vars must be set before `ollama serve` starts. Accept that the check is best-effort, not authoritative. Document this limitation.
