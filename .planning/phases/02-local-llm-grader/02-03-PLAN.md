<objective>
Fix the prefix match logic bug in checkOllamaAvailability and add test coverage.

Purpose: The model name matching in checkOllamaAvailability has an operator precedence bug (VERIFICATION.md anti-pattern). When a user specifies a model without a tag (e.g., "qwen3"), the prefix match branch is unreachable because `&&` binds tighter than `||`. This means `checkOllamaAvailability('qwen3')` would report "model not pulled" even when `qwen3:latest` is present in the Ollama tags list. The fix restores the intended behavior described in the code comment (Pitfall 4 from RESEARCH.md).

Output: Fixed `src/graders/index.ts` line 209 with correct operator precedence; new test cases in `tests/ollama-grader.test.ts` covering the prefix match path.

From src/graders/index.ts line 205-210 (BUGGY):
```typescript
const modelFound = models.some((m: any) => {
    const name: string = m.name || '';

    // Exact match or prefix match (e.g., "qwen3:4b" matches "qwen3:4b", "qwen3" matches "qwen3:latest")
    return name === model || name.split(':')[0] === model.split(':')[0] && name === model;
});
```

Bug explanation: `&&` binds tighter than `||`, so the expression evaluates as:
  `name === model || (name.split(':')[0] === model.split(':')[0] && name === model)`
The second branch requires BOTH prefix match AND exact match, which is redundant.
The prefix match path (user says "qwen3", tags has "qwen3:latest") is unreachable.

**Step 3 (GREEN): Fix the bug in src/graders/index.ts**

Change line 209 from:
```typescript
return name === model || name.split(':')[0] === model.split(':')[0] && name === model;
```

To:
```typescript
return name === model || (name.split(':')[0] === model.split(':')[0] && !model.includes(':'));
```

Logic: Match if (a) exact match, OR (b) the base model names match AND the user did not specify a tag. When the user specifies a tag (e.g., "qwen3:4b"), only exact match is used. When the user specifies just a model family (e.g., "qwen3"), any available tag for that family matches.

Update the comment on line 208 to match the new logic:
```typescript
// Exact match, or prefix match when user omits tag (e.g., "qwen3" matches "qwen3:latest")
```
