<objective>
Fix the LLM grader so it produces actual Ollama inference scores within the 60-second budget, and surface failure details in eval output when scores are zero.

Purpose: The bootstrap end-to-end test shows llm_rubric scoring 0.00 despite Ollama running. Three compounding issues cause this: (1) hardcoded 300s timeout masks the real cause, (2) no explicit num_ctx set — Ollama defaults to 2048 tokens (NOT the model's native 32K), which may silently truncate the grading prompt (~825 tokens) + response budget (2048 tokens) = ~2900 tokens needed, (3) zero scores are printed without explanation, hiding why grading failed. Note: qwen3:4b's native context is 32,768 tokens (with YaRN extension up to 131K), but Ollama's API default is 2048 regardless of model capability.

Output: src/types.ts with timeout_ms and num_ctx GraderConfig fields; src/graders/index.ts using 60s default timeout and num_ctx 4096 (enough for grading prompts, avoids Ollama's 2048 truncation default); src/evalRunner.ts printing grader details when score is 0.
</objective>

<task type="auto">
  <name>Task 2: Fix callOllama timeout and add num_ctx to API call</name>
  <files>src/graders/index.ts</files>
  <action>
In src/graders/index.ts, update the callOllama private method (currently at line ~222):

1. Change AbortSignal.timeout from the hardcoded `300000` (5 min) to `config.timeout_ms ?? 60000` (1 min default). The 5-minute value was the agent trial budget; grading a single completed response should take at most 60 seconds.

2. Add `num_ctx` to the Ollama options object in the request body. Use `config.num_ctx ?? 4096`. This is a correctness fix, not just a performance optimization: Ollama defaults num_ctx to 2048 (NOT the model's native 32,768), which silently truncates the grading prompt (~825 tokens) + response budget (2048 num_predict) = ~2900 tokens needed. At the 2048 default, the prompt may be truncated from the beginning with no error or warning in the API response. Setting 4096 gives comfortable headroom for the superlint_demo grading prompt while keeping memory/compute reasonable on CPU-only hardware.

3. Add a comment near the model selection (line ~223) noting that qwen3:4b is a thinking model: chain-of-thought tokens consume num_predict budget before the JSON answer. Users on CPU-only hardware should prefer a non-thinking model like phi3.5:3.8b via the `model` field in task.toml.

The updated fetch body should look like:

```typescript
body: JSON.stringify({
    model,
    prompt,
    stream: false,
    options: {
        temperature: 0,
        num_predict: 2048,
        num_ctx: config.num_ctx ?? 4096,
    },
}),
signal: AbortSignal.timeout(config.timeout_ms ?? 60000),
```

Do NOT change the retry logic, fallback chain, or parseResponse method.
  </action>
