# Phase 5.1: Tune OpenCodeAgent for Multi-Step Tool Execution - Research

**Researched:** 2026-03-11
**Domain:** Small LLM agentic tool calling via opencode CLI + Ollama
**Confidence:** HIGH (opencode internals verified from source), MEDIUM (model selection based on cross-referenced benchmarks)

## Summary

Phase 5 Plan 03 proved the OpenCodeAgent infrastructure works but the model could not complete a 3-step task through opencode's tool protocol. Deep source-level investigation of opencode's codebase reveals three actionable levers: (1) the permission system can **remove tools from the LLM's view** by setting them to `"deny"`, reducing the 10-tool overwhelm to ~5 essential tools; (2) Qwen 3.5 has **critical, confirmed tool-calling bugs on Ollama** (wrong format, unclosed think tags, missing generation prompt) that are still OPEN as of v0.17.7 -- this is very likely the primary root cause of our failures; (3) Qwen3:8B scored 0.933 F1 in Docker's practical tool-calling benchmark, outperforming every other local model except Qwen3:14B.

The recommended strategy is: first, switch from Qwen 3.5 to Qwen3 (which uses the stable Hermes-style tool calling format on Ollama), aggressively reduce the tool set via opencode permissions, and test with both 4B and 8B variants. The Qwen 3.5 tool calling bugs make it unreliable until Ollama ships fixes -- Qwen3 is the stable alternative. If Qwen3:4b cannot complete the 3-step task, escalate to Qwen3:8b (0.933 F1 on Docker benchmark) with careful thread management to avoid the CPU-hogging observed in Phase 4.

**Primary recommendation:** Switch to Qwen3:8b (or 4b as first attempt) with aggressive tool denial in opencode.json, reducing visible tools from 10 to ~5. Qwen 3.5's broken Ollama tool calling is very likely the root cause of Phase 5 Plan 03 failures.

## Critical Finding: Qwen 3.5 Tool Calling is Broken on Ollama

**Confidence: HIGH** -- verified from Ollama GitHub issues #14493 and #14745, with source-level analysis

Qwen 3.5 has **four confirmed, still-OPEN bugs** in Ollama's tool calling pipeline:

| Bug | Impact | Status (v0.17.7) |
|-----|--------|-------------------|
| Wrong tool call format (Hermes JSON instead of Qwen3-Coder XML) | All tool calls use format model was NOT trained on | **OPEN** |
| Repetition penalties silently ignored (Go runner) | Model loops endlessly, no `presence_penalty` effect | **OPEN** |
| Unclosed `</think>` tag in renderer | Corrupts multi-turn tool calling prompts | **OPEN** (parser fixed, renderer still broken) |
| Missing generation prompt after tool call turns | Breaks entire tool call round-trip loop | **OPEN** |

The registry config sets `renderer: "qwen3.5"` / `parser: "qwen3.5"`, which maps to the Qwen 3 Hermes-style JSON pipeline. But Qwen 3.5 was trained on the Qwen3-Coder XML format (`<function=name><parameter=key>value</parameter></function>`). The correct pipeline (`Qwen3CoderRenderer`/`Qwen3CoderParser`) exists but is wired to `"qwen3-coder"` only.

**This very likely explains Phase 5 Plan 03 failures.** The model was receiving prompt formats it was never trained on. Multiple users report the same behavior with opencode specifically: tool calls get printed as text instead of being executed.

**Sources:**
- [Ollama issue #14493](https://github.com/ollama/ollama/issues/14493) -- Detailed source-level bug report with 4 bugs
- [Ollama issue #14745](https://github.com/ollama/ollama/issues/14745) -- Qwen 3.5 9b with OpenCode prints tool calls instead of executing

## Opencode Internals (Source-Level Analysis)

**Confidence: HIGH** -- verified by reading opencode source code (sst/opencode repo, v1.2.24)

### Tool Set Restriction via Permissions

**KEY FINDING:** The permission system controls tool *visibility* to the LLM, not just execution gating.

In `packages/opencode/src/session/llm.ts` (line 258-266):
```typescript
async function resolveTools(input) {
    const disabled = PermissionNext.disabled(Object.keys(input.tools), input.agent.permission)
    for (const tool of Object.keys(input.tools)) {
        if (input.user.tools?.[tool] === false || disabled.has(tool)) {
            delete input.tools[tool]  // REMOVED from model's tool definitions
        }
    }
    return input.tools
}
```

The `disabled()` function in `permission/next.ts` (line 248-258) returns tools where the matching rule has `pattern: "*"` and `action: "deny"`. Setting a tool to `"deny"` in the opencode.json permission block **removes it entirely from the LLM's tool definitions**.

### Automatic Run Mode Denials

`opencode run` (non-interactive mode) automatically denies three additional tools via session-level rules:
- `question` (denied at line 358-362)
- `plan_enter` (denied at line 363-367)
- `plan_exit` (denied at line 368-372)

These are already handled. Permission requests in run mode are auto-rejected (line 544-556).

### System Prompt

For non-recognized models (including all Ollama models), opencode uses `packages/opencode/src/session/prompt/qwen.txt` (109 lines). Key characteristics:
- Very verbose: detailed instructions about conciseness, code style, proactiveness, tool usage policy
- Encourages Task tool usage: "prefer to use the Task tool in order to reduce context usage"
- Encourages parallel tool calls: "send a single message with multiple tools calls to run the calls in parallel"
- Strongly discourages text output: "Keep your responses short... fewer than 4 lines"

This prompt is **not ideal for small models** -- it's designed for Claude/GPT-class models. The verbosity consumes context, and the Task tool encouragement is counterproductive when we deny Task.

The system prompt is assembled in `packages/opencode/src/session/prompt.ts` (line 654-658):
1. Environment info (working directory, platform, date)
2. Skills info (if skill tool is not denied -- line 60 in system.ts)
3. Instruction files (from `.opencode/` directories)

When skill is denied via permissions, the skills system prompt section is **skipped entirely** (line 60 of system.ts).

### Agent Steps Configuration

The `steps` field on agent config controls max iterations (`agent.steps ?? Infinity` at prompt.ts:560). When `step >= maxSteps`, tools are disabled and the model receives a "max steps reached" message.

For our use case, the default (Infinity) is fine since we have the evalRunner's timeout as outer protection.

### SIGABRT Issue #13367 Status

The issue was reported for linux-arm64 (Ubuntu 24.04 aarch64) with kernel `6.8.0-1042-nvidia-64k`. The likely root cause is 64KB kernel page size vs opencode's binary compiled for 4KB page size (same issue as Claude Code #17941). No fix found in opencode commits as of the latest shallow clone. The issue remains **OPEN**.

However, for our Docker setup (WSL2-backed Docker Desktop on Windows ARM64), the kernel page size is 4KB (standard Linux kernel), so this specific issue should NOT affect us. Testing in Docker is still recommended to verify.

### Tracing Ollama API Requests

Ollama's built-in `OLLAMA_DEBUG=true` does NOT dump request/response payloads -- only HTTP status and URIs. Options for tracing:

1. **HTTP proxy**: Use mitmproxy between opencode and Ollama to capture full request/response bodies
2. **OLLAMA_DEBUG=true**: Basic server config logging only
3. **curl -v**: Manual API testing to see tool definitions
4. **opencode `--format json`**: Outputs structured events (tool uses, text, steps) to stdout

The most practical approach for debugging tool calls is `opencode run --format json "..." 2>&1` which emits JSON events for each tool call and response.

## Architecture Patterns

### Recommended opencode.json Configuration

```json
{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "ollama": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "Ollama (local)",
      "options": {
        "baseURL": "http://localhost:11434/v1"
      },
      "models": {
        "qwen3-8b-opencode-agent": {
          "name": "Qwen3 8B OpenCode Agent",
          "tools": true
        }
      }
    }
  },
  "model": "ollama/qwen3-8b-opencode-agent",
  "permission": {
    "read": "allow",
    "edit": "allow",
    "bash": "allow",
    "write": "allow",
    "glob": "deny",
    "grep": "deny",
    "list": "deny",
    "task": "deny",
    "todowrite": "deny",
    "skill": "deny",
    "webfetch": "deny",
    "websearch": "deny",
    "codesearch": "deny",
    "external_directory": "deny",
    "doom_loop": "allow"
  }
}
```

This configuration:
- Denies 8 tools, leaving only bash, read, edit, write (+ invalid internal)
- Denying `skill` also suppresses the skills system prompt section (saves tokens)
- Denying `task` prevents subagent spawning (which would fail anyway with a small model)
- `doom_loop: "allow"` lets the agent retry when it gets stuck (important for multi-step)

### Prompt Prefix Strategy

The current prefix is:
```
IMPORTANT: You MUST use the Bash tool to run ALL shell commands described below.
Do NOT stop after the first command. Execute EVERY command in order.
Do NOT summarize results between commands. Just run the next command immediately.
Do NOT use Skills or any feature other than Bash, Read, and Edit.
/no_think
```

Recommended changes:
1. **Remove Skill mention** -- when `skill` is denied, the model never sees it. Mentioning it wastes tokens.
2. **Be more explicit about the multi-step chain** -- small models need step-by-step reinforcement.
3. **Add "do not stop" reinforcement after each anticipated tool result** -- opencode's system prompt tells the model to "minimize output tokens" and "answer concisely", which may encourage early stopping.
4. **Keep `/no_think`** -- confirmed by Goose blog and Phase 4 findings that `/no_think` improves small model tool calling speed and reliability.

Recommended prefix:
```
CRITICAL: Execute ALL commands below in order. After each command completes, immediately run the next one. Do NOT stop, summarize, or explain between commands.
You have 4 tools: Bash, Read, Edit, Write. Use Bash for ALL shell commands.
After the last command, respond with a one-line summary.
/no_think
```

### Anti-Patterns to Avoid

- **Using Qwen 3.5 on Ollama**: Broken tool calling format (issue #14493). Switch to Qwen3.
- **Keeping all tools enabled**: 10+ tools overwhelms small models. Deny everything not needed.
- **Using thinking mode with small models for tool calling**: `/no_think` is faster and more reliable per Goose blog and Phase 4 findings.
- **Setting num_thread to all cores for 8B models**: Causes OS unresponsiveness. Use cores - 2 to cores / 2.
- **Using num_ctx 4096 with opencode**: The system prompt alone is ~2K tokens. With tool definitions and multi-turn history, 4096 is too tight. Use 8192.

## Common Pitfalls

### Pitfall 1: Qwen 3.5 Format Mismatch on Ollama
**What goes wrong:** Model produces tool calls that Ollama cannot parse, or prints them as text
**Why it happens:** Ollama uses Hermes JSON format for Qwen 3.5 but model was trained on Qwen3-Coder XML format
**How to avoid:** Use Qwen3 (stable Hermes format) instead of Qwen 3.5 until Ollama fixes land
**Warning signs:** Tool calls printed as `<tool_call><function=bash>...` text instead of being executed; agent stalls after thinking block

### Pitfall 2: Tool Count Overwhelm at Small Scale
**What goes wrong:** Model invokes wrong tools, uses Skill/Task instead of Bash, or generates malformed tool calls
**Why it happens:** Small models (3-8B) struggle with >5 tool definitions. Each tool adds ~100-200 tokens of schema.
**How to avoid:** Deny all unnecessary tools via opencode.json permissions
**Warning signs:** Model calls `skill` or `task` for basic operations; model generates `list` instead of `bash ls`

### Pitfall 3: Context Overflow with opencode's System Prompt
**What goes wrong:** Model output degrades or becomes nonsensical partway through the task
**Why it happens:** opencode's qwen.txt system prompt is ~110 lines + tool schemas (~500 tokens each) + conversation history
**How to avoid:** Use num_ctx 8192+ to ensure full system prompt + tool defs + 3-step conversation fits
**Warning signs:** Later tool calls become incoherent; model "forgets" earlier instructions

### Pitfall 4: 8B Model CPU Hogging on ARM64
**What goes wrong:** OS becomes unresponsive during model inference
**Why it happens:** Ollama's ARM64 thread detection may use ALL threads. 8B model at 8 threads on 12-core CPU = 100% utilization
**How to avoid:** Set `num_thread` explicitly in Modelfile (6 for 8B, 8 for 4B). Test interactively first.
**Warning signs:** Mouse lag, SSH disconnects, other processes freeze during inference

### Pitfall 5: Early Stopping After First Tool Call
**What goes wrong:** Model completes step 1 (superlint check) then generates a summary instead of continuing to steps 2-3
**Why it happens:** opencode's system prompt emphasizes brevity ("fewer than 4 lines", "minimize output tokens"). Small models interpret this as "stop after one action."
**How to avoid:** Strong prefix that explicitly says "execute ALL commands", "do NOT stop between commands". The `doom_loop: "allow"` permission helps opencode retry if the model stops prematurely.
**Warning signs:** Model outputs "Done!" or a summary after running only `superlint check`

   - What we know: Source code shows `delete input.tools[tool]` in resolveTools()

   - Recommendation: Verify with `--format json` or curl test before full experiments

3. **Does num_ctx 8192 fit in 16GB with 8B model + grader sequential loading?**
   - What we know: qwen3:8b is ~5GB at Q4_K_M. Grader uses qwen2.5:3b (~2GB). Sequential loading prevents overlap.
   - What's unclear: Exact KV cache memory at 8192 context for 8B model
   - Recommendation: Should fit (5GB model + 0.5-1GB KV cache << 16GB). Test on CI runner.

4. **Will opencode's linux-arm64 binary work in our Docker environment?**
   - What we know: SIGABRT issue #13367 affects 64KB page size kernels. WSL2 uses 4KB pages.
   - What's unclear: Whether there are other ARM64-specific issues
   - Recommendation: Test Docker ARM64 early in experiments. Fallback to x64 QEMU if needed.

5. **Is the user's locked decision "start with qwen3.5:4b" still optimal given the bug findings?**
   - What we know: Qwen 3.5 tool calling is fundamentally broken on Ollama (format mismatch)
   - Recommendation: Present findings to user. Research strongly recommends Qwen3 over Qwen 3.5. A quick Qwen 3.5 test can confirm the bug, then pivot to Qwen3.
