Verify that the Node.js environment is correctly configured for LocalProvider subprocess execution.

The user needs to verify their environment resolves this — either by configuring FNM for non-interactive shells or ensuring Node.js is on PATH for all bash invocations.

**Step 2: Identify the fix needed**

Common FNM scenarios on Windows:
1. **Git Bash**: FNM init is in `~/.bashrc` which only runs for interactive shells. Fix: add FNM eval to `~/.bash_profile` or `~/.profile` so non-interactive shells also get it.
2. **WSL2 Bash**: Similar — FNM init may only be in `.bashrc`. The LocalProvider uses `shell: 'bash'` which on Windows resolves to Git Bash, not WSL bash.
3. **PATH inheritance**: If the parent Node.js process (Claude Code) already has the correct PATH (because it was launched from an interactive shell), then `process.env.PATH` should contain FNM's Node.js. If it does, the subprocess should inherit it.

**Step 3: Apply environment fix**

Based on diagnosis, apply one of:
- Add FNM initialization to `~/.profile` (for non-interactive bash)
- Ensure Claude Code is launched from a shell that has FNM initialized
- Verify that the `bash` on PATH is Git Bash (not WSL bash) when running on Windows
