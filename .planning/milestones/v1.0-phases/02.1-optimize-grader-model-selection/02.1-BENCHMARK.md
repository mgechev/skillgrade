# Phase 2.1: Grader Model Benchmark Report

**Date:** 2026-03-09
**Platforms:** Snapdragon X Elite (32GB, 12 cores), ubuntu-24.04-arm CI (4 vCPU, 16GB)

## Summary

Eight models across three tuning profiles were benchmarked on a Snapdragon X Elite ARM64 laptop using direct Ollama API calls against three test fixtures (positive, empty, wrong). The benchmark measured wall time, tokens/second, JSON Schema compliance, and grading score accuracy.

**qwen2.5:3b** was selected as the default grader model based on its perfect score discrimination (positive=1.0, empty=0.0, wrong=0.0), fastest median wall time (4.6s on default profile), 100% JSON Schema validity, and small 2GB memory footprint. While phi3.5 variants scored 1.0 on all fixtures (failing to discriminate bad sessions), and llama3.2:3b scored 0.8 on negative cases (too lenient), qwen2.5:3b was the only model that correctly gave 0.0 to both empty and wrong submissions while giving 1.0 to the correct submission.

## Model Selection

**Selected model:** `qwen2.5:3b`
**Rationale:** Perfect discrimination across all fixtures -- the only model that correctly identified both negative cases (empty=0.0, wrong=0.0) while scoring the positive case 1.0. This is the critical quality for a grading model: distinguishing good work from bad. Its 4.6s median wall time on default profile (fastest among all models) and 100% JSON Schema compliance make it the clear winner.

## Tuning Impact

Using qwen2.5:3b median wall times (positive fixture, excluding cold-start run 1):

| Profile Transition | Default (ms) | Target (ms) | Change | Notes |
|-------------------|-------------|-------------|--------|-------|
| Default -> Optimized Env | 4,620 | 4,729 | +2.4% | Env vars have negligible impact on qwen2.5:3b (already fast) |
| Optimized Env -> Optimized All | 4,729 | 5,595 | +18.3% | API params (num_ctx=8192) add overhead vs Ollama's default 2048 |
| Default -> Optimized All (combined) | 4,620 | 5,595 | +21.1% | Larger context window costs time but prevents prompt truncation |

**Key finding:** For qwen2.5:3b specifically, tuning does not improve speed -- the model is already fast enough that the overhead of a larger context window dominates. The env vars (flash attention, quantized KV cache) have more impact on larger models. The `num_ctx=8192` increase from Ollama's default 2048 is necessary for correctness (prevents silent prompt truncation) even though it adds latency.

For phi3.5 models, the optimized-all profile caused significant slowdowns (65s median for q4_K_M positive fixture vs 12.8s on default), with two runs exceeding the 60s timeout. This confirms phi3.5 is poorly suited for the larger context window.
